# V1-V5 大版本服务范围说明

## 1. 文档定位

本文档用于统一说明 `V1` 至 `V5` 各次大版本更新在第一章课题中的服务范围、主要目标与边界。

本文档的作用不是重复每个子版本的全部执行细节，而是回答以下问题：

- 每个大版本主要在解决什么问题
- 每个大版本服务于第一章的哪一段主线
- 各版本之间是什么递进关系
- 当前为什么应从 `V4` 过渡到 `V5`

## 2. 第一章总体递进逻辑

第一章当前的整体推进顺序可以概括为：

1. 先建立数据基础与拼接边界
2. 再固定三层分析结构与主表
3. 再建立第三层 `PWS-year` 全国正式主线
4. 再补强真正第二层 `facility-month` 的高信息增强/机制支撑模块
5. 最后再做整体框架整合与泛化性外推审计

在这一逻辑下，`V1` 到 `V5` 的版本职责并不是并列的，而是逐步递进的。

## 3. 各大版本服务范围

### 3.1 `V1`

版本定位：

- 严格样本级保守基线与可拼接性审计阶段

主要服务范围：

- 围绕 `TTHM` 与核心化学变量进行严格样本级对齐
- 识别原始 `SYR4` 跨表拼接在最严格口径下的可用性边界
- 形成保守版 Spearman 基线与第一层 sanity check 结果

核心价值：

- 证明严格样本级下信息重合度有限
- 为后续不再把第一层当主建模底表提供证据

不承担的任务：

- 不承担正式多变量主建模
- 不承担全国尺度正式主模型

### 3.2 `V2`

版本定位：

- 第一章第一部分的数据基础审计与分析层级设计阶段

主要服务范围：

- 梳理 `DBP` 研究范围
- 解释核心变量环境意义
- 明确核心表之间的关系
- 审计三层级数据可用性与拼接条件
- 固定第一章分层研究设计

核心价值：

- 正式建立“第一层 / 第二层 / 第三层”的三层分析结构
- 明确第一章应先做数据基础与边界，而不是直接进入模型竞赛

不承担的任务：

- 不直接构建正式机器学习输入表
- 不直接产出正式主模型

### 3.3 `V3`

版本定位：

- 第二层与第三层原型主表落地阶段

主要服务范围：

- 生成 `V3_facility_month_master.csv`
- 生成 `V3_pws_year_master.csv`
- 固定字段来源、聚合规则、缺失语义、样本分层与禁止误用规则

核心价值：

- 把 `V2` 的层级设计真正落地为可复用的数据产品
- 为后续第三层机器学习输入层与第二层通路探索提供底座

不承担的任务：

- 不直接完成正式模型训练
- 不直接回答通路框架问题

### 3.4 `V3.5`

版本定位：

- 第三层 `PWS-year` 机器学习输入层冻结阶段

主要服务范围：

- 从第三层原型主表派生 `V4_pws_year_ml_ready.csv`
- 固定主结果变量、标签规则、缺失标记与样本分层
- 固定第三层内部“第一级样本 / 第二级样本 / 第三级样本”的命名与规则

核心价值：

- 为 `V4` 第三层正式建模准备可直接使用的输入底表
- 把第三层正式主线与高信息子样本实验的样本边界固定下来

不承担的任务：

- 不直接代表第二层 `facility-month` 的正式实验
- 不直接回答整体框架是否已经闭合

### 3.5 `V4`

版本定位：

- 第三层 `PWS-year` 正式主线构建与信息通路框架初步成型阶段

主要服务范围：

- 构建第三层全国正式主模型
- 完成 `baseline`、`structural`、`treatment` 的逐步增强链
- 在第三层内部第二级样本上检验 `pH + alkalinity + TOC` 水质增强路径
- 通过 `V4.7` 检验系统背景通路与水质特征通路的互补关系
- 把项目目标从“单一万能模型”收束为“分层预测框架”

核心价值：

- 明确第三层 `PWS-year` 第一级样本上的 `baseline + structural + treatment` 是当前全国正式主模型方向
- 明确第三层内部第二级样本上的 `baseline + pH + alkalinity + TOC` 是当前最核心的高信息水质增强证据
- 明确 `V4.7` 是高信息样本中的整合证据，而不是新的全国正式主模型

边界说明：

- `V4` 的核心任务是第三层，不是第二层 `facility-month` 的正式通路构建
- `V4` 中涉及的“第二级样本”属于第三层 `PWS-year` 内部样本分层，不等于第一章的第二层数据

### 3.6 `V5`

版本定位：

- 真正第二层 `facility-month` 高信息增强 / 机制支撑模块构建阶段

主要服务范围：

- 回到第一章真正的第二层 `facility-month`
- 审计第二层候选变量覆盖率、重合度与可执行边界
- 固定第二层 baseline 与候选增强链
- 做一轮受控的第二层正式实验
- 判断第二层能否形成独立成立的高信息增强 / 机制支撑模块

核心价值：

- 将“第二层有机制价值”从概念判断推进到正式模块构建
- 回答第二层与第三层内部第二级样本到底有何不同
- 为后续整体框架整合提供真正属于第二层的数据证据

边界说明：

- `V5` 不是无边界寻找更多新通路
- `V5` 的重点是验证并构建第二层模块，而不是进行大规模性能优化

## 4. 当前为什么应从 `V4` 进入 `V5`

当前从 `V4` 进入 `V5` 的原因主要有三点：

- 第三层 `PWS-year` 正式主模型主线已经基本成型，继续在 `V4` 内部做局部优化的收益开始下降
- 当前所谓高信息水质增强证据，主要仍建立在第三层内部第二级样本上，尚未真正转化为第二层 `facility-month` 的正式模块
- 如果不单独进入 `V5`，后续第一章容易继续把“第二层数据”和“第三层内部高信息子样本”混写

因此，`V5` 的作用不是简单新增一个版本号，而是把真正第二层的任务独立出来，避免第一章后续逻辑继续混乱。

## 5. 当前版本关系的一句话总结

当前 `V1` 到 `V5` 的递进关系可以概括为：

- `V1` 解决严格样本级边界
- `V2` 解决数据基础与分层设计
- `V3` 解决主表落地
- `V3.5` 解决第三层输入层冻结
- `V4` 解决第三层正式主线与信息通路初步成型
- `V5` 将继续解决真正第二层 `facility-month` 模块是否能够独立成立的问题
