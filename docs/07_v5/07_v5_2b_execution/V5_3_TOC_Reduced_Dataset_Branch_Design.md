# V5.3 TOC Reduced Dataset 专题分支设计

- 更新时间：2026-04-10 10:05（Asia/Hong_Kong）
- 对应阶段：`V5.2b`
- 目标阶段：`V5.3`

## 1. 文档定位

本文档不是 `V5.3` 的正式执行报告，也不是启动 `TOC` 建模的直接指令。

本文档的作用是把原本容易被误写成“主链默认下一步”的 `V5.3`，正式改写为：

- 第二层 `TOC` reduced dataset 专题分支

也就是说，`V5.3` 当前应先被设计成一个边界收紧、样本受限、目标明确的专题任务，而不是第二层主链的自动延伸。

## 2. 为什么 `V5.3` 不能直接沿用原始线性设想

原因来自两个前序结论：

### 2.1 `V5.0` 已经给出的结构性约束

真正第二层下：

- `TTHM + pH + alkalinity + TOC` strict complete-case 只有 `37` 行

这意味着：

- `TOC` 一旦接到 `pH + alkalinity` 主链后面，样本会立刻塌缩到极小规模

### 2.2 `V5.2` 没有给出足够强的主链稳定证据

`V5.2` 结论是：

- `test` 端有正向信号
- `validation PR-AUC` 没有同步改善

因此，当前不具备把第二层直接写成“稳定扩变量链”的证据基础。

## 3. `V5.3` 的正式重定义

`V5.3` 当前应重定义为：

- `TOC reduced dataset branch`

它的核心问题不再是：

- “在第二层正式主链上继续做下一轮默认增量”

而应改成：

- “在真正第二层上，`TOC` 是否还能以专题 reduced dataset 的形式保留补充价值”

## 4. `V5.3` 应回答的核心问题

如果后续正式进入 `V5.3`，至少应回答以下问题：

1. `TOC` 在真正第二层下能否构成一个可解释、可复现、样本规模可接受的专题子数据集
2. `TOC` 的价值更接近机制补充证据，还是仅仅来自极小样本上的偶然波动
3. `TOC` 专题结果能否为第二层“有限高信息增强”提供附加支撑
4. `TOC` 是否应该继续保留为第二层模块的一部分，还是应明确停在 `V5.2`

## 5. `V5.3` 的建议任务边界

### 5.1 应做的事

- 先重新审计 `TOC` 专题可执行性
- 明确专题样本规模、正例数、split 后类别分布
- 明确是采用严格 `complete-case`，还是只做专题审计后直接收束
- 与 `V5.2` 做边界清楚的衔接说明

### 5.2 不应做的事

- 不应把 `TOC` 直接写成第二层正式主链默认下一步
- 不应把 `TOC` 结果误写成第二层增强主模型成立
- 不应在 `V5.3` 中顺手继续带入 `free_chlorine` 或 `total_chlorine`
- 不应为追求结果数字而放松第二层对象与解释边界

## 6. `V5.3` 的推荐执行顺序

如果之后要正式做 `V5.3`，推荐顺序是：

1. 新增 `V5.3` 专题 prompt 或协议文档
2. 先做 `TOC reduced dataset` 审计
3. 如果审计后仍确认有价值，再决定是否进入真正的小样本建模
4. 若样本与 split 稳定性不足，则直接以专题收束结论结束 `V5.3`

## 7. `V5.3` 与整体 `V5` 的关系

重定义后的 `V5.3` 不承担“继续证明第二层主链成熟”的任务。

它承担的是更受限的任务：

- 判断 `TOC` 是否值得作为第二层专题补充分支保留

因此它在整体 `V5` 中的角色应是：

- 可选专题分支

而不是：

- 默认主链延长段

## 8. 当前建议

基于 `V5.2` 当前结果，最合理的建议是：

- 先完成 `V5.2b` 的制度与边界收束
- 暂不直接启动 `V5.3` 建模
- 如果后续需要进入 `V5.3`，应严格按照“reduced dataset 专题分支”的方式重开任务

## 9. 本文档结论

`V5.3` 当前最合理的定义不是“第二层主链继续扩到 `TOC`”，而是“第二层 `TOC` reduced dataset 专题分支评估”。

这一定义更符合：

- `V5.0` 的可执行性审计结论
- `V5.2` 的弱正向但未完全稳定结果
- 第二层当前真实而受限的模块角色
