# V5.4 候选信息通路审计与框架组装协议

## 1. 文档定位

本文档用于固定 `V5.4` 更新的核心规则：在 `V5.3` 已明确框架面向真实监管决策支持之后，后续工作不应直接无边界扩展变量或堆叠模型，而应先建立一套候选信息通路的审计、准入、分级和框架组装协议。

本协议回答两个问题：

1. SYR4 中哪些信息通路有资格进入后续分层风险支持框架？
2. 进入框架的信息通路应如何组织，而不是被简单处理成“多个模型投票”？

本协议不是最终模型清单，也不是最终特征表。后续每条通路仍需经过覆盖率、标签重叠率、泄露风险、增量价值和稳定性审计后，才能确定是否进入正式框架。

## 2. V5.4 的研究边界

`V5.4` 的目标是制定规则，不是直接训练新模型。

当前阶段应避免三类误用：

- 不应因为某个字段“有数据”就默认进入模型。
- 不应因为某个通路有机制解释就忽略覆盖率和可执行性。
- 不应把每个通路机械训练成一个独立模型后再简单平均或投票。

后续 V5 更新应按本协议逐条审计候选通路，直到能够明确最终框架可以搭建哪些模型、每个模型的输入 `X` 是哪些、输出证据是什么、以及这些模型如何被整合成监管可用的决策支持结果。

## 3. 候选通路范围

结合当前 `SYR4` 数据和已构建的 V3 主表，候选信息通路至少包括：

| 通路 | 候选信息 | 初步定位 |
|---|---|---|
| 结构背景通路 | `system_type`、`source_water_type`、服务人口、系统规模等 | 主筛查模型候选 |
| 处理工艺通路 | 过滤、吸附、氧化、消毒、氯胺化、次氯酸盐化、处理工艺记录数、工艺名称列表等 | 主筛查或机制增强候选 |
| 设施复杂度通路 | `water_facility_type`、设施数量、供水设施数量、流量记录、设施类型列表等 | 主筛查或结构复杂度代理 |
| NOM/有机前体物通路 | `TOC`、`DOC`、`UV254`、`SUVA`、paired `TOC-Alkalinity` | 可选机制增强模块候选 |
| 酸碱与缓冲条件通路 | `pH`、`alkalinity` | 可选机制增强模块候选 |
| 消毒剂与残余消毒剂通路 | `free_chlorine`、`total_chlorine`、`chloramine`、残余氯、二氧化氯、CT、厂内消毒剂浓度等 | 可选机制增强模块候选 |
| 监测覆盖度与数据质量通路 | `source_module_count`、`n_core_vars_available`、`n_extended_vars_available`、`match_quality_tier`、`annual_match_quality_tier` 等 | 证据质量层候选 |
| DBP 组分与共现通路 | 单项 THM、单项 HAA、`bromate`、`chlorite` | 结果谱系/后续解释模块候选 |
| 微生物与消毒需求代理通路 | total coliform、E. coli、HPC、Giardia、Cryptosporidium、paired microbe-disinfectant residual 等 | 探索性专题候选 |
| 合规与纠正行动通路 | ADWR compliance、corrective actions、cryptobinning 等 | 探索性/制度背景候选，需严格防泄露 |
| 广义水化学背景通路 | phasechem、rads 等未系统整合模块 | 仅先做字段与覆盖率审计 |

上述清单是候选范围，不代表全部进入模型。

## 4. 通路准入规则

每条候选通路进入后续建模前，必须至少回答以下问题。

| 审计问题 | 含义 | 决策作用 |
|---|---|---|
| 是否有机制或监管解释？ | 该通路是否能解释 DBP 风险或监管决策意义 | 无解释时不进入核心框架 |
| 是否能在预测时合理获得？ | 决策者在真实使用时是否可能提前提供该信息 | 不能提前获得时不得作为预测输入 |
| 是否存在同周期目标泄露？ | 是否使用了同一时期、同一目标家族的结果变量 | 存在泄露时不得进入预测模型 |
| 字段覆盖率是否足够？ | 字段在候选样本中非缺失比例 | 决定是否可作为主模型输入 |
| 标签重叠覆盖率是否足够？ | 字段与 `TTHM/HAA5` 标签是否在同一分析单元中同时存在 | 决定是否可训练对应通路模型 |
| 应用覆盖率是否足够？ | 真实监管场景中用户是否大概率能提供该字段 | 决定是否能作为框架必需输入 |
| 是否相对基础模型有增量价值？ | 加入该通路后是否稳定改善排序、召回、校准或解释 | 决定是否保留为模型通路 |
| 是否跨年份、跨系统类型或跨水源类型稳定？ | 通路信号是否可能只是局部样本偶然结果 | 决定是否进入正式框架 |
| 是否能支持行动建议？ | 输出能否转化为补充监测、重点复核或局部调查建议 | 决定是否有决策支持价值 |

## 5. 覆盖率判定规则

本项目中的“覆盖率”不是模型准确率，而是某个通路在可用于建模或决策的样本中拥有有效数据的比例。

后续通路审计至少需要区分三类覆盖率：

| 覆盖率类型 | 定义 | 例子 |
|---|---|---|
| 字段覆盖率 | 某字段或通路本身在多少样本中非缺失 | 多少 `PWS-year` 有 `TOC` |
| 标签重叠覆盖率 | 有字段值的样本中，多少同时有 `TTHM/HAA5` 标签 | 多少样本同时有 `TOC` 与 `TTHM` 标签 |
| 应用覆盖率 | 真实使用时监管者或水厂是否大概率能提供该字段 | 环保局是否通常能提供 `SUVA/UV254` |

粗略判定规则如下：

| 覆盖率情况 | 建议定位 |
|---|---|
| 高覆盖，例如 `>70%` | 可考虑作为主筛查模型候选输入 |
| 中等覆盖，例如 `30%-70%` | 适合作为机制增强通路候选 |
| 低覆盖，例如 `5%-30%` | 适合作为高信息子模型或专题分析候选 |
| 极低覆盖，例如 `<5%` | 通常不适合建模，优先保留为描述性分析或暂缓 |

这些阈值不是硬性统计定理，而是后续审计时的工作判据。最终是否纳入，还需结合泄露风险、增量价值和稳定性判断。

## 6. 通路分级规则

审计完成后，每条通路应被分配到以下角色之一。

| 角色 | 含义 | 示例 |
|---|---|---|
| 主筛查模型输入 | 覆盖广、可提前获得、适合服务系统级排序 | 系统类型、水源类型、人口规模、部分处理工艺和设施复杂度特征 |
| 可选机制证据模块 | 机制意义强，但覆盖率或时空完整性有限 | `TOC/DOC/UV254/SUVA`、`pH/alkalinity`、消毒剂残留 |
| 证据质量模块 | 不解释为化学风险机制，但用于衡量证据完整性和可信度 | `source_module_count`、`n_core_vars_available`、`match_quality_tier` |
| 结果谱系或解释模块 | 更适合解释已观测风险构成，不适合直接预测同周期同目标 | 单项 THM/HAA 组分、`bromate/chlorite` |
| 探索性专题模块 | 机制或制度意义可能存在，但覆盖、时序或泄露风险未清楚 | 微生物、合规、纠正行动、phasechem、rads |
| 暂不纳入 | 缺少覆盖、解释、稳定性或存在严重泄露风险 | 同周期同目标结果字段、极低覆盖且不可解释字段 |

## 7. 模型组装规则

后续框架不应默认“一条通路一个模型”。是否合并或拆分，应按以下原则判断。

### 7.1 可以合并的情况

覆盖率相近、机制相近、应用场景相近的通路可以合并为同一个模型输入组。

例如：

- 结构背景、系统类型、水源类型、人口规模、部分处理工艺和设施复杂度，可合并为广覆盖主筛查模型。
- `pH`、`alkalinity`、`TOC`、`DOC`、`UV254`、`SUVA` 可在覆盖率允许时合并为 NOM/水质机制增强模块。

### 7.2 不宜强行合并的情况

覆盖率差异大、机制差异大、真实可获得性不同或泄露风险不同的通路不应强行合并。

例如：

- `SUVA/UV254` 如果覆盖率极低，不应作为主筛查模型必需输入。
- 监测覆盖度变量不应与化学机制变量混为同一种“风险原因”。
- 同周期 DBP 组分不应直接用于预测同周期同家族总量风险。
- 合规和纠正行动信息在未明确时间顺序前，不应与前置预测变量混合入模。

### 7.3 增量价值检验

通路是否保留，不应只看单独训练效果，而应看相对基础模型是否提供增量信息。

推荐比较链为：

1. 结构基础模型。
2. 结构基础模型 + 单一候选通路。
3. 结构基础模型 + 已确认稳定通路 + 新候选通路。
4. complete-case 敏感性检查。
5. 跨年份、跨系统类型或跨水源类型稳定性检查。

若某通路在机制上合理但无法稳定提供增量信息，应保留为解释或审计结果，而不是强行进入正式模型。

## 8. 三层框架组装规则

最终框架不应是“多个模型简单投票”，而应组织为三层结构。

### 8.1 第一层：广覆盖主筛查模型

定位：框架的基础排序层。

输入：

- 结构背景
- 系统类型
- 水源类型
- 服务人口
- 处理工艺摘要
- 设施复杂度代理
- 非泄露的覆盖与结构信息

输出：

- 系统层或 `PWS-year` 层基础高风险评分
- 广覆盖风险排序

作用：

- 在信息有限但覆盖广的条件下，告诉监管者“哪些系统应优先关注”。

### 8.2 第二层：可选机制证据模块

定位：框架的机制增强层，不是主模型的替代品。

输入：

- 有机前体物信息，例如 `TOC/DOC/UV254/SUVA`
- `pH/alkalinity`
- 消毒剂与残余消毒剂信息
- 局部 `facility-month` 数据

输出：

- 前体物证据评分
- 酸碱/缓冲条件证据评分
- 消毒剂暴露证据评分
- 局部月份或设施证据评分

作用：

- 在额外数据存在时，把系统从“结构上可疑”推进到“机制证据也支持可疑”。
- 在额外数据不完整时，允许模块跳过，而不是让整个框架失效。

### 8.3 第三层：证据整合与行动建议层

定位：最终决策输出层。

输入：

- 主筛查模型分数
- 可选机制模块分数
- 数据完整性
- 模型适用范围
- 缺失情况
- 启用与未启用的证据模块

输出：

- 综合风险评分
- 综合风险等级
- 证据等级
- 优先级排序
- 启用的证据来源
- 缺失的关键证据
- 建议监管行动
- 是否建议进入后续因果分析模块

建议行动可包括：

- 常规跟踪
- 优先补充监测
- 重点复核
- 局部月份或设施调查
- 进入后续因果分析候选模块

## 9. 泄露控制规则

后续所有通路审计和建模都必须遵守以下泄露控制规则：

- 不得使用同一时期、同一目标家族的结果变量预测该目标。
- 不能用同月单项 THM 组分预测同月 `TTHM` 高风险。
- 不能用同月单项 HAA 组分预测同月 `HAA5` 高风险。
- 不能用同年 `tthm_*` 摘要字段预测同年 `TTHM` 高风险。
- 不能在未明确时间先后的情况下，将纠正行动、合规状态或事后监管响应作为前置预测输入。
- 可将 DBP 组分用于结果谱系解释、交叉风险探索或严格滞后特征设计，但必须单独说明用途和边界。

## 10. 后续 V5 执行计划

基于本协议，后续 V5 更新建议按以下顺序推进：

1. 完成候选通路字段清单与当前 V3 主表字段映射。
2. 对结构背景、处理工艺、设施复杂度、NOM 前体物、酸碱与缓冲条件、消毒剂、监测覆盖度等已进入 V3 主表的通路做优先审计。
3. 对尚未充分整合的原始 SYR4 模块做覆盖率与泄露风险盘点，包括单项 DBP、`bromate/chlorite`、微生物、合规/纠正行动、phasechem、rads 等。
4. 为每条通路给出正式归属：主筛查模型输入、可选机制证据模块、证据质量模块、结果谱系/解释模块、探索性专题模块或暂不纳入。
5. 在完成通路准入判定后，再确定最终框架中可以训练哪些模型、每个模型的输入 `X` 是哪些、输出证据是什么。
6. 最后再搭建证据整合与行动建议层，避免先堆模型后解释。

## 11. 当前结论

`V5.4` 的核心结论是：

本项目后续不应继续以“发现一个新变量就训练一个新模型”的方式推进，而应先固定候选信息通路审计与框架组装协议。只有通过机制解释、覆盖率、标签重叠、真实应用可获得性、泄露风险、增量价值和稳定性审计的通路，才能进入正式框架。

最终框架应被组织为：

1. 广覆盖主筛查模型。
2. 可选机制证据模块。
3. 证据整合与行动建议层。

这种结构能更准确地服务环保局或水厂的实际问题：在不完整监管数据条件下，先对系统进行风险排序，再在高信息数据存在时补充机制证据，最后给出可执行的监管行动建议。
