# V5 总体计划

## 1. 文档定位

本文档用于固定 `V5` 阶段的总体目标、研究问题、更新拆分、输入输出与停止边界。

`V5` 的核心任务不是继续在 `V4` 第三层结果上做局部性能优化，也不是无边界扩展更多信息通路，而是回到第一章真正的第二层 `facility-month`，判断该层级能否形成一个独立成立的高信息增强 / 机制支撑模块。

## 2. V5 的正式定位

`V5` 的正式定位为：

- 第二层 `facility-month` 高信息增强 / 机制支撑模块构建阶段

这一定义包含三层含义：

- `V5` 研究对象是第一章真正的第二层 `facility-month`，而不是第三层 `PWS-year` 内部第二级样本
- `V5` 的目标是验证并构建第二层模块，而不是预设第二层一定已经成熟成立
- `V5` 更强调变量覆盖边界、可执行性与模块角色定位，而不是单纯追求性能数字

## 3. V5 需要回答的核心问题

`V5` 当前至少需要回答以下三个核心问题：

1. 第二层 `facility-month` 是否能形成一个比第三层内部第二级样本更接近环境过程的水质增强模型？
2. 在第二层里，哪些变量组合是可执行的、稳定的，而不是只在极少数完整样本上成立？
3. 第二层最终应承担“机制支撑线”“高信息增强线”，还是两者兼有？

## 4. V5 的总体原则

### 4.1 先审计，再建模

`V5` 不应一开始就直接跑模型，而应先完成变量覆盖率、重合度与可执行性审计。

### 4.2 先建立边界，再讨论性能

当前第二层最重要的问题不是性能最优，而是：

- 哪些变量真实可用
- 哪些组合可训练
- 哪些结论值得保留为正式模块

### 4.3 保持逐步增强链

第二层实验也应优先采用逐步增强链，而不是一开始就做大而全整合。

### 4.4 明确和第三层内部第二级样本的区别

`V5` 必须持续明确：

- 第二层 `facility-month` 与第三层 `PWS-year` 内部第二级样本是不同对象
- 两者都可能承载高信息证据，但研究角色并不相同

## 5. V5 的建议拆分

### 5.1 `V5.0` 第二层候选变量审计

目标：

- 固定第二层这一轮要回答的问题
- 审计候选变量覆盖率、pairwise 重合度与 complete-case 可执行性
- 识别哪些变量值得进入下一轮正式实验

重点检查变量：

- `pH`
- `alkalinity`
- `TOC`
- `free_chlorine`
- `total_chlorine`

重点输出：

- 第二层候选变量覆盖率报告
- pairwise 重合度摘要
- complete-case 风险说明
- 第二层 baseline 候选与增强候选清单

停止边界：

- 如果变量覆盖率与重合度不足以支撑正式实验，应在此阶段就明确收束，而不是强行进入建模

截至 `2026-04-08` 的 `V5.0` 审计结论已明确：

- 第二层 `baseline` 可以继续推进
- 第二层第一轮正式增强应固定为 `baseline + pH + alkalinity`
- 真正第二层下的 `baseline + pH + alkalinity + TOC` strict complete-case 只剩 `37` 行，不能再被默认写成下一步必做正式主链
- `free_chlorine` 与 `total_chlorine` 当前都应暂停进入第二层正式主链

### 5.2 `V5.1` 第二层 baseline 固定

目标：

- 固定第二层正式 baseline
- 固定样本口径、标签口径、切分口径与禁止误用规则

重点输出：

- 第二层 baseline 方案文档
- 第二层 baseline 首轮结果
- 第二层样本与特征边界说明

停止边界：

- 若 baseline 尚不稳定，不应继续推进增强变量实验

### 5.3 `V5.2` 第二层机制核心 stage1

目标：

- 在第二层上先测试 `baseline + pH + alkalinity`
- 判断这组最基础的机制核心组合是否具有稳定价值

重点输出：

- `baseline + pH + alkalinity` 正式实验结果
- missing pattern 与 complete-case 的对照判断
- 第二层机制增强是否初步成立的第一轮判断

停止边界：

- 若 `pH + alkalinity` 在第二层完全无法形成稳定增益，则需重新评估第二层的正式角色

### 5.4 `V5.3` 框架决策支持定义

目标：

- 在 `V5.2` 与 `V5.2b` 之后，正式固定整个 `V5` 线的现实使用场景、输入输出逻辑与框架价值边界
- 明确该框架服务的不是对完整结果的重复计算，而是在不完整监管数据条件下进行风险筛查、优先级排序与局部证据整合
- 明确第三层是广覆盖主筛查层，第二层是局部场景支撑层

重点输出：

- 框架决策支持定义文档
- 框架输入输出示例文档
- 第三层与第二层在最终框架中的角色说明
- 面向监管者和水厂的使用语义说明

停止边界：

- 若框架最终仍只能重述已完整观测到的监管结果，则应进一步收缩目标，不应过度宣称预测价值

### 5.4b `V5.2b` 第二层制度收束与专题分支重定义

目标：

- 在 `V5.2` 后正式收束第二层当前样本制度、模块边界与结论边界
- 明确 `V5` 当前尚未形成 `V4` 式正式三级样本制度
- 将 `V5.3` 从默认线性主链下一步，改写为 `TOC reduced dataset` 专题分支

重点输出：

- 第二层样本制度与边界总结文档
- `V5.3 TOC reduced dataset` 专题分支设计文档
- `V5.2` 之后的后续行动计划

停止边界：

- 若 `V5.2` 只能支持弱正向且受限的机制支撑证据，则不应继续把第二层写成稳定增强主链

### 5.4c `V5.3x` TOC reduced dataset 专题子分支

目标：

- 在 `V5.3` 框架定义固定后，再决定是否单独启动 `TOC reduced dataset` 专题子分支
- 判断 `TOC` 是否还值得作为第二层受限专题补充证据保留

重点输出：

- `TOC` 专题审计结果或受限小样本结果
- 与 `V5.3` 框架主输出语义之间的衔接说明

停止边界：

- 若样本与 split 稳定性不足，则应直接收束为专题边界结论

### 5.5 `V5.4` 候选信息通路审计与框架组装协议

目标：

- 在 `V5.3` 已固定框架决策支持定位后，先建立候选信息通路的审计、准入、分级和框架组装规则
- 明确后续 V5 更新不应直接无边界扩变量，也不应把每个通路机械训练成一个独立模型后简单投票
- 固定三层框架组装逻辑：广覆盖主筛查模型、可选机制证据模块、证据整合与行动建议层
- 为后续逐条审计 SYR4 候选通路提供统一判断标准

重点输出：

- 候选信息通路审计与框架组装协议文档
- 通路准入规则，包括机制解释、覆盖率、标签重叠覆盖率、真实应用可获得性、目标泄露风险、增量价值和稳定性
- 通路分级规则，包括主筛查模型输入、可选机制证据模块、证据质量模块、结果谱系/解释模块、探索性专题模块和暂不纳入
- 模型组装规则，明确哪些通路可合并、哪些通路应作为可选模块、哪些通路只能用于证据质量或后续解释
- 三层框架组装规则，明确最终输出应包括综合风险等级、证据等级、优先级排序、建议行动和因果分析后续候选

停止边界：

- 到本阶段结束时，应形成后续 V5 通路审计的统一协议，但不应提前写死最终模型清单或最终特征表
- 任一通路只有在后续审计中通过覆盖率、标签重叠、泄露风险和增量价值判断后，才能进入正式框架

### 5.6 `V5.5+` 候选通路逐条审计与模型准入判定

目标：

- 按照 `V5.4` 协议逐条审计 SYR4 候选信息通路
- 明确每条通路最终是否进入框架、进入哪一层、输入 `X` 是哪些、输出证据是什么
- 在通路审计完成后，再确定最终框架中可以训练哪些模型以及如何整合

重点输出：

- 候选通路字段清单与 V3 主表字段映射
- 每条通路的覆盖率、标签重叠覆盖率、应用覆盖率、泄露风险和增量价值审计结果
- 每条通路的最终归属：主筛查模型输入、可选机制证据模块、证据质量模块、结果谱系/解释模块、探索性专题模块或暂不纳入
- 最终框架可训练模型清单、各模型输入特征 `X` 与输出证据定义

停止边界：

- 若某条通路覆盖率不足、同周期目标泄露风险无法排除、或相对基础模型无稳定增量价值，则不应进入正式预测模型
- 通路审计不能无限延长，每条通路都应给出“进入、可选、仅解释、暂缓或排除”的明确结论

## 6. V5 第一轮建议变量链

当前经 `V5.0` 审计后调整为：

1. `baseline`
2. `baseline + pH + alkalinity`
3. `baseline + pH + alkalinity + TOC` 仅在 `V5.2` 后仍确认值得保留时，才作为专题分支重新评估

补充变量原则：

- `free_chlorine` 仅在第二层覆盖率和重合度明显优于第三层时再考虑
- `total_chlorine` 只有在审计明确显示具有实际可训练价值时才允许进入
- 不建议在 `V5` 初期直接做大而全整合模型

## 7. V5 的主要输入

`V5` 当前主要依赖以下输入：

- 第二层原型主表 `V3_facility_month_master.csv`
- 既有第二层相关字段字典与构建说明
- `V4` 已形成的第三层正式主模型结论
- 第三层内部第二级样本上的水质增强与 `V4.7` 整合证据

## 8. V5 的主要输出

`V5` 结束后，理想情况下应至少产出：

- 第二层变量覆盖率与可执行性审计报告
- 第二层 baseline 与增量实验结果
- 第二层框架使用场景、输入输出与决策支持定义
- 第二层模块总结文档
- 第二层与第三层内部第二级样本的差异说明
- 可直接接入第一章总框架的正式中文表述

## 9. V5 与后续整体框架的关系

`V5` 的作用不是取代 `V4`，而是补齐当前框架中尚未独立成型的一块。

当 `V5` 基本完成后，第一章整体框架应能够更清晰地写成：

- 第一层：严格审计层
- 第二层：真正的 `facility-month` 高信息增强 / 机制支撑层
- 第三层：`PWS-year` 全国正式主模型层
- 第三层内部第二级样本：高信息子样本验证与整合证据层
- `V4.7`：信息通路互补证据

在此之后，再考虑：

- 总框架整合
- 论文统一写法
- 泛化性外推审计

## 10. 当前阶段的一句话总结

`V5` 的任务不是继续无边界寻找更多新通路，而是回到真正的第二层 `facility-month`，用审计先行、逐步增强、边界清晰的方式，判断第二层是否能够形成一个独立成立的高信息增强 / 机制支撑模块，并将其接入第一章已经初步成型的分层预测框架。
