# 第一章第一部分：DBP 数据基础、可拼接性审计与后续分析层级设计

## 1. 第一章第一部分的目标定位

第一章第一部分的核心目标，不是直接建立预测模型，也不是直接开展因果识别，而是为后续研究建立可信的数据产品、变量体系与方法边界。具体而言，本部分要完成四项基础性工作：第一，界定 SYR4 中与 DBP 研究直接相关的数据范围；第二，解释核心结果变量、前体变量、反应条件变量和消毒过程变量的环境意义；第三，厘清核心数据表的结构、键关系与合理 join 粒度；第四，通过分层审计明确后续机器学习主线与机制/受约束因果分析主线分别应建立在哪一种数据层级上。

这一部分不能等同于建模，原因在于：SYR4 的原始监测数据并非为研究设计而采集，缺失既严重又具有明显的非随机性；不同变量分布在不同 occurrence 表中，样本级重合度极低；处理工艺信息和结构性背景信息位于独立的 treatment 子库中，天然不与单条样本结果一一对应。如果在未完成数据层级辨析和可拼接性审计之前直接进入模型训练，很容易将“数据缺失结构”误当作“环境机制”，也容易将不同粒度表之间的机械拼接误写成科学关联。

因此，本部分最终应形成以下成果：一份 DBP 变量环境意义框架；一份核心数据表关系说明；一份严格样本级、设施-月份级、系统-年份级三层级数据可用性审计；一份对既有 V1 严格样本级工作的正式学术定位；以及一份区分机器学习主表与机制分析主表的后续数据路线图。它与后续 ML / 因果分析的关系，不是替代这些工作，而是为它们提供“数据契约”：明确哪些变量可以在何种层级使用、哪些问题适合在何种层级回答、哪些推断边界不能被跨越。

| 氯胺 | `SYR4_Disinfectant Residuals/CHLORAMINE (1006).csv` | 消毒过程变量 | `MG/L` | 用于识别以氯胺为主的消毒语境 | 有助于解释不同消毒体系下 THM/HAA 的形成路径差异 | 机制分析、分层比较 |

从环境过程上看，`TOC / DOC / SUVA / UV254` 共同构成“前体与有机物性质”模块，`pH / 总碱度` 构成“反应环境”模块，`游离余氯 / 总氯 / 氯胺` 构成“消毒暴露”模块，而 `TTHM / HAA5` 是最终结果模块。后续任何机制分析都应尽量按照这三个解释模块组织，而不是把所有变量简单当作无差别特征堆叠进模型。

## 3. 数据表关系框架

### 3.1 核心表的结构定位

| 数据表/子库 | 一行代表什么 | 主键/候选键 | 更合理的连接方式 | 说明 |
| --- | --- | --- | --- | --- |
| `SYR4_THMs` | 单次 THM 监测结果记录 | 候选键为 `PWSID + WATER_FACILITY_ID + SAMPLING_POINT_ID + SAMPLE_COLLECTION_DATE`，但表内可能有重复，需要先源内聚合 | 严格样本级或先聚合到设施-月份级/系统-年份级 | `TTHM` 是结果主表，4 个 THM 组分可用于解释总量与组分关系 |
| `SYR4_HAAs` | 单次 HAA 监测结果记录 | 同上 | 与 THMs 平行，推荐先独立聚合后再在设施-月份级或系统-年份级对照 | `HAA5` 不应被当作 TTHM 的附属变量 |
| `SYR4_DBP_Related Parameters` | 单次 DBP 前体或反应条件测定记录 | occurrence 文件同样使用四键候选键；`Paired TOC-Alkalinity` 是例外，为设施-月份 reduced dataset | occurrence 文件可做严格样本级审计；`Paired TOC-Alkalinity` 只能在设施-月份级接入 | 这一子库内部本身就包含两种不同数据逻辑 |
| `SYR4_Disinfectant Residuals` | 单次消毒残留监测记录 | occurrence 四键候选键 | DBP 主线更适合聚合到设施-月份级；微生物主线才更接近样本级配对 | residual 变量在 DBP 与微生物主线中服务的粒度不同 |
| `syr4_treatment` | system-facility-plant-process-flow 结构与工艺信息 | 主键依表而异，核心桥接字段为 `Water System ID`、`Water Facility ID` | 先将结果层聚合到设施或系统层，再把 treatment 当作解释层接入 | 它不是结果表，而是结构/工艺解释数据库 |

### 3.2 关键 join 判断

第一，所谓“严格样本级 join”，应限定为 `PWSID + WATER_FACILITY_ID + SAMPLING_POINT_ID + SAMPLE_COLLECTION_DATE` 四键对齐，并且要先在各 occurrence 表内对重复键做源内聚合，避免把同源重复行误写成跨表样本量。

第二，`SYR4_THMs`、`SYR4_HAAs`、`SYR4_DBP_Related Parameters` 中的 occurrence 文件和 `SYR4_Disinfectant Residuals` 都属于原始结果层，因此在数据库形态上是并列关系，不存在谁天然是“主表”。研究者必须根据问题先选定分析层级，再决定哪张表作结果主线。

第三，`Paired TOC-Alkalinity.csv` 不是 occurrence 表，而是设施-月份 reduced dataset。本次审计显示，它共有 `92,666` 个设施-月份键，但与 TTHM 设施-月份键的交集仅 `306` 个，与 HAA5 的交集仅 `140` 个。这说明它可以作为专题化补充数据来源，但不能承担通用主表职责，更不能按采样日期去和单条 THM/HAA 记录强行一对一拼接。

第四，`syr4_treatment` 必须被理解为独立的关系子库，而不是一张普通宽表。其合理用法是：先把 occurrence 结果聚合到设施或系统层，再通过 `Water Facility ID` 和 `Water System ID` 将工艺、plant、flow 等结构信息作为解释层接入。它不适合在严格样本级被机械拼接到单条监测记录上。

### 3.3 可直接写入论文的核心表关系概述

可将本研究涉及的 SYR4 DBP 数据结构概括为三层：第一层是以 THMs、HAAs、DBP-related occurrence 和 disinfectant residuals 为代表的原始监测结果层，其基本单位是单次监测记录；第二层是以 `Paired TOC-Alkalinity` 为代表的专题化 reduced dataset 层，其基本单位已下沉为设施-月份；第三层是以 `syr4_treatment` 为代表的结构与工艺解释层，其作用不是提供结果变量，而是为设施和系统层分析提供背景解释。由此决定了本研究不能采用统一的“单表思维”，而必须采用“先分层聚合、再跨层连接”的策略。

## 4. 三层级数据可用性与方法判断

### 4.1 审计结果摘要

本次基于 `data_local/V2_Chapter1_Part1_DBP_Data_Foundation/` 下的审计结果，对三种分析层级进行了正式比较。核心结果如下：

| 分析层级 | TTHM 唯一键数 | HAA5 唯一键数 | TTHM + 核心四变量至少 2 个预测变量 | TTHM + 核心四变量全齐 | HAA5 + 核心四变量全齐 | 方法判断 |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| 严格样本级 | 927,683 | 818,802 | 1,601 | 0 | 0 | 适合保守基线、样本级可拼接性审计、sanity check，不适合作为主建模底表 |
| 设施-月份级 | 549,730 | 481,761 | 3,811 | 0 | 0 | pairwise 重合度明显改善，适合机制分析起点，但仍不支持“完整多变量底表”假设 |
| 系统-年份级 | 199,802 | 165,379 | 26,975 | 60 | 60 | 覆盖最广，适合全国机器学习主表，但牺牲时间与设施内异质性 |

补充看变量与 TTHM 的交集，可以看到三个极其重要的事实。

第一，严格样本级下，尽管 `pH` 本身有 `570,790` 个非缺失严格样本键、`TOC` 有 `421,597` 个、游离余氯有 `1,649,118` 个，但真正与 TTHM 严格重合的键数分别只有 `6,918`、`1,968` 和 `2,560`。这说明“各表内部样本量很大”不等于“跨表后仍然可用”。

第二，设施-月份级确实改善了 pairwise 重合度。例如，与 TTHM 的交集从样本级的 `6,918 / 981 / 1,968 / 2,560`（pH / 碱度 / TOC / 游离余氯）上升到 `9,796 / 3,220 / 2,824 / 5,446`。但即便如此，`TTHM + pH + 碱度 + TOC + 游离余氯` 仍然没有形成任何一个完整同时非缺失的设施-月份单元。因此，设施-月份级适合做模块化机制分析、变量对子分析和受约束时间窗口分析，不适合被描述为“已经获得完整多变量主表”。

第三，系统-年份级通过更强的时间和空间聚合显著提高了覆盖范围。此时与 TTHM 相交的 `pH`、`总碱度`、`TOC` 键数分别达到 `30,968`、`33,721` 和 `18,750`，`TTHM + 核心四变量全齐` 首次出现 `60` 个系统-年份单元。这说明全国尺度预测主表更现实的选择确实是系统-年份级；但同时也意味着该层级不再适合承担细粒度机理解释任务。

### 4.2 为什么严格样本级不适合作为主建模底表

严格样本级的价值在于“最保守”，而它不适合作为主建模底表，原因至少有三点。第一，它衡量的是最严苛口径下的跨表重合度，而不是最大化可用信息；第二，它把监管监测中的时间错位和非同步采样全部视为不可用，从而天然压缩了前体、反应条件和结果变量的共同样本；第三，这种极低重合度会使模型训练更多反映“谁被同步监测过”，而不是“哪些环境因素稳定驱动了 DBP”。因此，严格样本级应被保留为可拼接性审计和 sanity check，而不是主模型训练底表。

### 4.3 设施-月份级与系统-年份级的优缺点比较

设施-月份级的优势是更贴近操作单元和短期运行环境，保留了设施内差异与季节性信息，也更符合将前体、反应条件和消毒残留放在同一运行背景下解释 DBP 的逻辑。它的不足是，在当前实际覆盖结构下，仍然不能自然形成“全变量完整”的主表，因此更适合小而精的机制变量集、pairwise 框架、窗口化匹配和受约束子样本分析。

系统-年份级的优势是覆盖最广，更便于形成全国可比的系统画像，也更方便接入 `SYSTEM_TYPE`、`SOURCE_WATER_TYPE`、人口规模和系统级结构变量。它的不足是会把设施差异、月份差异和瞬时消毒环境折叠进年尺度统计中，因此适合回答“哪些系统更可能高风险”“哪些系统特征与较高 DBP 负荷相关”这类全国画像问题，但不适合直接承担细粒度机理解释。

### 4.4 HAA5 如何纳入

HAA5 不应被视为 TTHM 完成后的附属扩展，而应从一开始就与 TTHM 采用同一套三层级框架。当前审计显示，HAA5 在严格样本级、设施-月份级和系统-年份级分别有 `818,802`、`481,761` 和 `165,379` 个非缺失唯一键，且与 TTHM 的交集覆盖率在三个层级分别为 `93.40%`、`96.89%` 和 `97.27%`。这说明 TTHM 与 HAA5 具有较高并行分析可行性，后续主表设计应尽量把两者作为平行结果变量，而不是一先一后割裂处理。

## 5. 对 V1 的正式定位与评价

现有 `V1_TTHM_strict_spearman_base_data` 的正式定位，应被界定为“严格样本级保守基线、样本级可拼接性审计与第一层 sanity check”。它的首要目标不是给出最优预测性能，而是回答一个更基础的问题：在 `PWSID + WATER_FACILITY_ID + SAMPLING_POINT_ID + SAMPLE_COLLECTION_DATE` 的最保守匹配规则下，TTHM 与核心前体/反应条件/消毒过程变量究竟能够重合到什么程度。

这项工作的价值至少有四点。第一，它构建了可审计的严格样本级母表，明确了重复键聚合、原值保留与轻度清洗的规则；第二，它证明了在最严格样本口径下，跨表重合度极为有限，从而为后续“为什么必须转向更高层级”提供了证据；第三，它给出了第一层相关方向的 sanity check，即 `TOC` 与 `TTHM` 中等正相关、`总碱度` 与 `TTHM` 中等负相关、`游离余氯` 与 `TTHM` 弱负相关、`pH` 与 `TTHM` 相关很弱；第四，它为后续设施-月份级和系统-年份级分析提供了一个不能被覆盖回写的保守对照基线。

但 V1 也有明确局限。它以严格四键同步观测为前提，而监管数据中大量“真实但不同步”的信息会因此被排除；它当前围绕 TTHM 主线构建，不能代表 HAA5 的完整数据路线；它更适合做变量方向验证和拼接性审计，而不能直接升级成主建模数据底表。尤其关键的是，V1 已经显示严格样本级下 `TTHM + pH + 总碱度 + TOC + 游离余氯` 完整样本为 `0`，这已经足以说明它不能承担后续全国主模型或多变量机理主表的任务。

论文中对 V1 的推荐表述可以写成：本研究首先构建严格样本级 V1 基线数据产品，用于在最保守的四键对齐规则下审计不同 DBP 相关监测表之间的可拼接性，并对关键变量与 TTHM 的相关方向进行第一层 sanity check。该基线用于界定数据边界与方法边界，而不承担后续主模型训练底表的职责。

## 6. 后续 ML / 因果分析的数据路线图

### 6.1 A 线：全国广覆盖主线（机器学习主表）

建议分析单元为 `PWS-year`。

之所以优先选择这一层级，是因为全国机器学习主线首先要解决的是覆盖广度、可比性和稳定性问题，而不是尽量保留局部瞬时过程细节。系统-年份级在当前审计中拥有 `199,802` 个 TTHM 键、`165,379` 个 HAA5 键，并且已经能够形成一定规模的多变量共同非缺失单元，因此是最现实的全国主表起点。

这一层级建议保留的变量包括：年度 `TTHM`/`HAA5` 汇总指标，系统类型、原水类型、服务人口、设施数量或设施类型摘要、可获得的 treatment 结构摘要、年度 pH/碱度/TOC 统计量、年度游离余氯/总氯统计量，以及各变量对应的样本数与覆盖率指标。聚合方式应以解释性和稳健性为优先，例如对结果变量保留年均值、中位数、P90、最大值、超标比例、样本数；对前体和过程变量保留均值、中位数、最大值、IQR 或标准差、非缺失样本数；对类别变量保留主类型或 one-hot 摘要。这样构建出的系统-年份主表，适合回答“哪些系统更容易处于高 DBP 水平”“哪些系统级背景因素与较高 DBP 风险相关”“全国尺度上风险系统的空间与结构画像如何”等问题。

这一主线不宜被描述为因果分析底表，因为它通过强聚合换取了覆盖率，代价是设施内、月份内和采样时点差异都被折叠掉了。

### 6.2 B 线：高信息机制主线（机制分析 / 受约束因果分析）

建议分析单元为 `PWS-facility-month`。

这一层级更适合机制分析，是因为 DBP 形成更接近设施运行条件和较短时间窗口，而不是纯粹的系统年度平均状态。设施-月份级仍然保留了设施差异和季节性，能比系统-年份级更接近“同一设施在相近运行背景下的前体-反应-结果关系”。但当前审计也表明，它并不能自然支持“全变量完整表”的想象，因此该层级应被设计成“高信息机制子样本框架”，而不是“全国统一多变量底表”。

时间窗口匹配建议采用分层规则。第一步，先以 `PWSID + WATER_FACILITY_ID + year + month` 建立月度骨架，不再做样本级硬拼。第二步，在同一设施同一月份内分别对 TTHM/HAA5、pH、碱度、TOC、游离余氯、总氯等变量做月度汇总，优先保留均值、中位数、最大值和样本数。第三步，如需更接近机制过程，可在设施-月份层内进一步构建“同月值”和“前一月值”两个版本，分别用于同步机制分析和简化滞后分析，但必须明确其口径差异。第四步，仅在高信息子样本中接入 `Paired TOC-Alkalinity`、treatment process 或 plant 信息，作为专题化解释层，而不是强制要求所有设施都具备。

这一层级建议保留的变量包括：月度 TTHM/HAA5 汇总值，月度 TOC、pH、总碱度、游离余氯、总氯统计量，必要时补充氯胺、DOC、SUVA、UV254，外加设施类型、原水类型、treatment process 标记以及该月的样本数。它适合回答的问题包括：在同一设施或相似设施中，哪些前体/反应环境组合更容易对应较高 TTHM/HAA5；不同消毒体系下 THM/HAA 的形成方向是否不同；季节性与设施类型如何影响 DBP 形成背景。它不能简单外推到全国，是因为能进入高信息机制子样本的设施本身就不是随机样本，覆盖结构与全国总体存在系统偏差。

## 7. 论文可直接使用的详细提纲

### 7.1 研究对象与数据范围

1. 说明 SYR4 作为全国尺度真实监管监测数据库的优势与局限。
2. 明确本部分仅聚焦 DBP 主线，不扩展到 PFAS、砷、总大肠菌群等其他主题。
3. 界定结果变量、前体变量、反应条件变量、消毒过程变量和结构变量的范围。

### 7.2 DBP 相关变量的环境意义

1. 分别解释 TTHM、HAA5 的监管意义与作为结果变量的角色。
2. 解释 TOC、DOC、SUVA、UV254 所代表的前体总量与有机物性质信息。
3. 解释 pH、总碱度所代表的反应环境。
4. 解释游离余氯、总氯、氯胺所代表的消毒暴露语境。

### 7.3 核心数据表结构与关系

1. 说明 THM/HAA/DBP-related occurrence/disinfectant residuals 属于并列的原始结果层。
2. 说明 `Paired TOC-Alkalinity` 是设施-月份 reduced dataset，而非样本表。
3. 说明 `syr4_treatment` 是结构与工艺解释层，不是结果表。
4. 给出严格样本级、设施-月份级、系统-年份级各自的合理 join 方式。

### 7.4 不同分析层级的数据可用性比较

1. 展示严格样本级可拼接性极低但最保守。
2. 展示设施-月份级 pairwise 重合度改善，但仍不足以支撑完整多变量底表。
3. 展示系统-年份级覆盖最广，但牺牲细粒度过程信息。
4. 比较 TTHM 与 HAA5 在三层级中的并行可用性。

### 7.5 第一阶段数据基础结论

1. V1 严格样本级工作应被定位为保守基线和拼接性审计。
2. 后续全国机器学习主线应以 `PWS-year` 为主表。
3. 后续高信息机制主线应以 `PWS-facility-month` 为主表。
4. DOC、SUVA、UV254 目前更适合作为专题化补充变量，而不是主表默认变量。

## 8. 可执行任务清单

1. 先固定三层级口径与字段字典。  
输出：三层级键定义、变量清单、统一字段说明。  
作用：避免后续 ML 与机制分析使用不同命名和不同粒度。

2. 基于 `PWS-year` 构建全国主表原型。  
输出：系统-年份级结果变量汇总表、前体/反应条件/残留年度汇总表、结构变量表。  
作用：为后续全国机器学习和风险画像提供主输入。

3. 基于 `PWS-facility-month` 构建高信息机制表原型。  
输出：设施-月份级骨架表、同月变量汇总、必要的滞后版本。  
作用：为后续机制分析和受约束因果分析做准备。

4. 把 treatment 信息改造成设施层解释变量。  
输出：facility-process、facility-plant、system-facility 的可接入摘要表。  
作用：为全国主表和机制主表补充结构与工艺背景。

5. 设计 TTHM 与 HAA5 的平行结果任务。  
输出：TTHM/HAA5 双结果变量的统一分析框架。  
作用：保证两条 DBP 主线在变量、层级和方法边界上保持一致。

6. 为机制主线定义“核心小变量集”。  
输出：一版优先变量集，例如 `TOC + pH + 总碱度 + 游离余氯 + 总氯`，以及专题增强变量集，例如 `DOC + SUVA + UV254`。  
作用：避免在覆盖率不足时直接追求全变量完备。

7. 在论文中正式写入 V1 的定位说明。  
输出：V1 的方法描述、价值说明和局限说明段落。  
作用：防止把样本级基线误写成主建模底表。

8. 在进入任何 ML 或因果分析之前，先完成主表版本冻结。  
输出：`PWS-year` v1 与 `PWS-facility-month` v1 的字段清单、构建规则和覆盖率摘要。  
作用：确保后续分析建立在稳定、可复现的数据产品之上。
